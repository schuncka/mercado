VirtualBOSS.modulo_USUARIO
Por vezes denominados internautas, utilizadores ou usuários ( "users"), são identificações únicas (User ID - Identificação do usuário) que, em conjunto com uma senha, permitem a utilização do sistema por qualquer indivíduo.

Este cadastro de usuário faz o vínculo a uma ENTIDADE específica de origem, tendo como exemplos de entidades os cadastros de FORNECEDORES, CLIENTES ou COLABORADORES. Também podemos chamar esta abstração de "Conta de Usuário".

Um COLABORADOR, para utilizar o sistema, necessita um USUÁRIO cadastrado relacionado desta forma aos seus dados de ENTIDADE (no caso: colaborador). Um cadastro de usuário é uma identificação única chamada de USER ID (ou simplesmente ID), que é o seu "apelido" que, junto a sua senha de acesso, lhe permitirá "logar" (efetuar "login", ou seja, ter acesso ao VirtualBOSS) para utilização do sistema. Cada conta de usuário possui dados específicos do mesmo, como: e-mail, nome, além dos principais USER_ID e SENHA que são utilizados para identificá-lo dentro do sistema e relacionar suas permissões de acesso a cada operação com direito disponibilizado por um módulo.
Grupos de usuários:
O controle das permissões dos usuários se dá através dos direitos sobre as operações de cada aplicação, de forma mais individual e abrangente possível. A classificação de usuários em GRUPOS é utilizada para diferenciá-los em conjuntos maiores definindo modo de abertura/operação do sistema, telas e interface para cada caso:
  • NORMAL: Grupo padrão dos usuários em geral. Mais indicado ser usado para contas de usuários de COLABORADORES.

  • CLIENTE: Todo usuário que é relativo a um elemento da ENTIDADE CLIENTE (ou mesmo fornecedor) indica ao sistema que este deve entrar em modo HELPDESK, pemitindo que este usuário, que representa o seu CLIENTE, opere dentro destas características do sistema visualizando, criando e interagindo com chamados de dúvidas, sugestões, problemas e alterações.

  • MANAGER: Grupo especial para contas de usuários COLABORADORES que tenham uma característica de GERENCIAMENTO, pois independente da parte de direitos, usuários deste grupo tem acesso e visualizam funcionalidades especiais em alguns módulos onde considera-se pertinente (* listadas abaixo).

  • SUPERUSER(SU): Recomendado que exista apenas uma conta de usuário deste tipo e é dado acesso a ela a modificações em banco de Dados e outras características. O sistema é entregue com apenas um usuário SU (_athenas) que serve, dentre outras coisas, para que a nossa equipe de suporte possa ter acesso ao sistema a fim de melhor identificar uma ocorrência.

Nota: Lista módulos e funcionalidades/características específicas no tratamento de usuários do grupo MANAGER:
modulo_ACCOUNT, modulo_AGENDA, modulo_AVISO_CFG, modulo_BS (ATIVIDADE), modulo_CATEGORIAS, modulo_CHAMADO, modulo_HORARIO, modulo_TODOLIST (TAREFAS) , modulo_PROCESSO e modulo_USUARIO.

Conceito de Aplicação/Módulo

Considerando a palavra "Aplicação" como sinônima de "Módulo": onde determinado grupo de ações sobre uma entidade, conjunto de tabelas ou qualquer abstração de cadastros pode ser realizada, temos neste sistema diversos módulos acessíveis aos usuários cadastrados no sistema, relacionando estas ações com permissões de operação e acesso as mesmas que podem ser configuradas neste nesta aplicação corrente, para cada usuário.

Cada módulo possui um conjunto de ações que são passíveis de se registrar um direito de execução sobre ela para cada usuário cadastrado, onde podemos resumir o conceito em Ações e Direitos, onde:

"ação" é uma operação da aplicação e "direito" é a permissão de executar uma determinada ação.


A aplicação e suas ações

Uma aplicação (módulo) possui operações que são passíveis de controle de permissões, exemplo: este módulo de USUÁRIOS possui as operações de INS, UPD, DEL, COPY, UPD_DIR e VIEW (Inserção, Alteração, Deleção, Cópia, Alteração de Direitos e Visualização). Desta forma, para cada conta de usuário, é possível definir quais operações ele pode ou não realizar neste ou em qualquer outro módulo determinado.

O usuário e seus direitos

Cada usuário tem ou não a ação relativa a aplicação, ou seja, uma vez que o módulo de usuários tem estas operações INS, UPD, DEL, COPY, UPD_DIR e VIEW, cada usuário deverá ter ou não permissão em cada uma delas dentro desta aplicação. Exemplo:

USER_ID......: fulano
APLICAÇÃO.: modulo_usuario (operações: INS, UPD, DEL, COPY, UPD_DIR e VIEW)
DIREITOS....: INS=TRUE//UPD=TRUE//DEL=FALSE//COPY=FALSE//UPD_DIR=FALSE//VIEW=TRUE
CAMPOS, "REGRA de NEGÓCIO" e FEATURES

Dentre os campos existentes para o cadastro ao qual este módulo trabalha, destacamos abaixo apenas os de especial relevância, com relação a alguma "regra de negócio" ao qual tal campo se relacione, a saber:

ID Usuário: ID - identificador único do usuário em todo sistema (até 50 caracteres)
Apelido: Este campo determina o NOME VISÍVEL para o cliente externo (que pode abrir chamados).
Entidade/Código: Relaciona este usuário a uma entidade, por exemplo, Colaboradores, Clientes e/ou Fornecedores
Grupo: *citado acima no tópico grupos de usuários
Entrada: determina qual o painel inicial que este usuário deve visualizar ao entrar no sistema
Chamados de: determina quais chamados, ou melhor chamados "de quem" este usuário deve ver/atender - isto interfere diretamente na visualização fornecida pelo "Painel Padrão", onde os chamados dos clientes são apresentados para serem atendidos.


Regras de negócios

Este módulo se caracteriza por regras simples, mas ressaltamos que ele trabalha com a parte chave do sistema, ou seja, seus USUÁRIOS e a tudo que ele se relaciona.


Observação: LOGIN de SU - é possível efetuar login no sistema com qualquer usuário desde que com uma senha de um usuário SU, logo, recomendamos fortemente que o sistema seja configurado com poucos SUPERS SU e que suas senhas sejam de altíssima segurança (e pouco prováveis de usuários comuns terem ou até mesmo a possibilidade de não se ter usuários SU, conforme o caso, somente o usuário SU que nós, os fornecedores do sistema temos para poder dar suporte ao mesmo.



Features

Além das funções básicas de qualquer módulo do sistema - Inserir, Alterar, Remover e Pesquisar (consideramos um módulo como composto de forma abstrata sempre por estes elementos: dialog(s), filtro(s) e grade de dados), podemos ter features extras conforme o caso:

1. "menu combo".
  • Clonar direitos: permite a clonagem (cópia) de direitos de um usuário para outro.
  • Clonar atalhos: permite a clonagem (cópia) dos atalhos exibidos na barra superior do "Painel Padrão".
2. "ícones de ação" da grade de dados:
  • Configurar DIREITOS: acesso a dialog de confioguração de direitos do usuário em questão.
  • Copiar usuário: permite a cópia de um usuário, juntamente com seus direitos de acesso.
